## Business Context

**Client:** 3Sisters Cleaning and Home Organizing (NYC) — cliente REAL em produção no Launch27 hoje. Migração planejada para xcleaners com cutover alvo Semana 7.

**Why this story first:** Todas as Stories 1.2-1.6 de Epic A dependem do pricing engine. Sem pricing funcional, não há booking válido, não há preview, não há migration possível. Esta é a única story blocking em Epic A.

**Non-negotiable constraint:** O motor de pricing precisa replicar os cálculos de Launch27 com tolerância ±$0.01 em 10 bookings reais do histórico 3Sisters. Qualquer divergência maior quebra a confiança da Ana (owner 3Sisters) e trava o cutover.

## User Story

**As** Ana Afonso (owner da 3Sisters Cleaning, Tenant/Admin),
**I want** um motor de pricing híbrido que calcule bookings com fórmula configurável (base + quartos + banheiros × tier + extras − desconto de frequência ± ajuste manual + imposto) E permita override por entry específica quando eu quiser customizar preço,
**So that** eu possa migrar da Launch27 para xcleaners mantendo exatamente os mesmos preços que meus customers pagam hoje, sem ter que re-digitar 32 services manualmente e sem risco de cobrar valor diferente do Launch27.

**Example booking real (3Sisters Launch27) que o engine deve replicar:**
```
Service (2 Rooms × 1 Bathroom, Basic) ..... $275.00
+ Extras (Stairs) ......................... $30.00
= Subtotal ................................ $305.00
− Discount (Weekly 15%) ................... $45.75
± Adjustment (manual) .................... −$29.58
= Amount before tax ...................... $229.67
+ Sales tax (4.50% NYC) .................. $10.34
= Final .................................. $240.01
```

### AC2 — Pricing Engine Module (`app/modules/cleaning/services/pricing_engine.py`)

**Must expose:**

```python
async def calculate_booking_price(
    business_id: UUID,
    service_id: UUID,
    tier: Literal['basic', 'deep', 'premium'],
    extras: list[dict],  # [{"extra_id": UUID, "qty": int}]
    frequency_id: UUID | None,
    adjustment_amount: Decimal = Decimal('0'),
    adjustment_reason: str | None = None,
    location_id: UUID | None = None,
    db: Database = ...,
) -> dict:
    """
    Returns complete pricing breakdown per ADR-001 Decision 2 schema.
    Used both by preview endpoint and booking confirmation.
    """
```

**Return value MUST match `price_snapshot` JSONB schema** defined in ADR-001 Seção Decision 2:
- `formula_id`, `service_id`, `tier`, `tier_multiplier`, `base_amount`, `bedrooms`, `bedroom_delta`, `bathrooms`, `bathroom_delta`, `override_applied` (bool), `subtotal_service`, `extras[]` (array with id/name/qty/price), `extras_sum`, `subtotal`, `frequency_id`, `frequency_name`, `discount_pct`, `discount_amount`, `adjustment_amount`, `adjustment_reason`, `amount_before_tax`, `tax_pct`, `tax_amount`, `final_amount`, `calculated_at`

**Canonical calculation order (must follow exactly):**
```
1. service_amount = override OR ((base + bedrooms × α + bathrooms × β) × tier_multiplier)
2. extras_amount  = Σ(extra.price × extra.qty)
3. subtotal       = service_amount + extras_amount
4. discount_amount = round(subtotal × discount_pct / 100, 2)
5. amount_before_tax = subtotal − discount_amount + adjustment_amount
6. tax_amount     = round(amount_before_tax × tax_pct / 100, 2)
7. final_amount   = amount_before_tax + tax_amount
```

**Override precedence:**
```
IF cleaning_service_overrides row exists WHERE service_id=? AND tier=? AND is_active=TRUE:
    service_amount = override.price_override  # flat, skip formula
ELSE:
    service_amount = formula-computed
```

**Tier multiplier applies ONLY to `(base + BR·α + BA·β)`, never to extras.** Extras are flat regardless of tier.

**Rounding:** use Python `Decimal` with `ROUND_HALF_UP` for money; convert to NUMERIC(10,2) only at final assignment. Track 4 decimal places internally.

**Sales tax lookup:** find most recent `cleaning_sales_taxes` row WHERE `location_id=?` AND `effective_date <= booking.scheduled_date` AND `is_archived=FALSE`.

**Error handling:**
- Missing formula for business → raise `PricingConfigError("No active formula for business")`
- Missing service → raise `PricingConfigError("Service not found")`
- Invalid tier for service → fallback to formula (no override exists for this tier)
- Missing sales tax config → use `tax_pct = 0` with warning log
- Invalid frequency_id → use `discount_pct = 0`

### AC6 — Snapshot Imutável

**Comportamento obrigatório:**
1. Ao criar booking com status `draft → scheduled`:
   - Chama `calculate_booking_price(...)`
   - Grava resultado completo em `cleaning_bookings.price_snapshot JSONB`
   - Cria rows em `cleaning_booking_extras` com `name_snapshot` e `price_snapshot` (preservando preço no momento da reserva mesmo se `cleaning_extras.price` mudar depois)
2. Ao editar booking posteriormente (ex: rescheduled, adjustment change):
   - `price_snapshot` **NÃO é recalculado automaticamente**
   - UI mostra botão explícito **"Recalculate pricing"** — se owner clica, avisa "This will overwrite the original price snapshot. Continue?" antes de chamar engine de novo
   - Log audit do recálculo (who, when, old_final vs new_final)
3. Bookings em status `completed`, `cancelled`, `no_show` são **READ-ONLY** no preço (sem recalculate).

```python
@pytest.mark.parametrize("fixture", LAUNCH27_BOOKINGS_10)
async def test_pricing_matches_launch27(fixture, db):
    result = await calculate_booking_price(**fixture.input, db=db)
    assert abs(result['final_amount'] - fixture.expected.final_amount) <= Decimal('0.01'), \
        f"Pricing mismatch: got {result['final_amount']}, expected {fixture.expected.final_amount}"
```

### Completion Notes List

**Decisions & tradeoffs:**

1. **`base_amount` field semantics when override active:** When override applies, `base_amount` in the snapshot reflects the override value (not the formula base). This deviates slightly from ADR-001 schema but preserves snapshot clarity ("this is what was charged"). Alternative: keep base_amount as formula value, add `override_amount` field — rejected for simplicity.
2. **Extras not in catalog:** If `extra_id` doesn't exist in `cleaning_extras` (e.g. archived), skipped with warning log (not hard failure). Rationale: resilience during migration period when Ana may archive extras post-booking.
3. **Tax fallback graceful (log + 0%):** Missing `cleaning_sales_taxes` row → warning + `tax_pct=0`, not error. Rationale: Dallas market has no tax config yet; blocking bookings would halt onboarding.
4. **`_parse_tier_multipliers` handles both dict and str:** asyncpg JSONB may arrive as either. Defensive parsing avoids production surprise.
5. **`breakdown_to_jsonb` helper provided** for persistence use case — not strictly required by tests but makes T6 (booking integration) easier.

**Known limitations (v1):**

- Only `price_override` field is granular in `cleaning_service_overrides` (no `duration_override`) — per ADR-001 Decision 3 v1 scope
- No memoization / cache — each call fetches all data from DB. Acceptable for current load; optimize in v2 if preview endpoint becomes hot path.
- `adjustment_amount` is not bounded — owner can enter any value (including |$10000|). Validation belongs in API layer (Task 3), not engine.

**Follow-ups raised:**

- **R4 (from migration validation):** services existing with NULL `tier`/`bedrooms`/`bathrooms` will fail formula path. Recommendation: add backfill script that sets `tier='basic', bedrooms=0, bathrooms=0` for all rows where `tier IS NULL`. OR add override per service during onboarding. Ana/Mario's 32 existing services need mapping session pre-cutover.
- **R9 (from migration validation):** `cleaning_client_schedules` (newer recurring table) does NOT have `frequency_id` FK. Only legacy `cleaning_recurring_schedules` was backfilled in migration 021. If `cleaning_client_schedules` is in active use, a migration 022 is needed. Requires verification.

Short-term mitigation: warn-log when `|calculated − agreed_price| > $0.50`. Long-term: migration 022 adds `frequency_id` to `cleaning_client_schedules` OR VARCHAR backfill. Empirical validation via C7 Ana cross-check (5 real bookings ±$0.01).
